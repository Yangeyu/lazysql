# ADR 0003：TUI 渲染框架 —— Ink（睁眼选择，非默认）

- 状态：已接受
- 日期：2026-06-24

## 背景

lazysql 的核心视图是**可滚动、可逐格编辑的数据网格**，这是对 TUI 框架最苛刻的场景。
需在「DX/生态成熟度」与「大网格渲染性能」之间权衡。

## 候选与实测要点

| 框架 | 网格性能 | DX/成熟度 | 与 Bun 契合 | 分发 |
|------|---------|----------|------------|------|
| **Ink (React)** | ⚠️ 无原生 ScrollView，需自研 windowing；**硬编码 ~30 FPS 上限**；简单应用 ~50MB | ✅ 最成熟、社区最大 | ✅ | native 依赖少 |
| OpenTUI (Zig 核心) | ✅ 原生帧 diff、无 FPS 上限、为大网格而生 | ⚠️ 年轻(2024+)、生态小 | ✅ 原生 | ⚠️ 多一个 Zig 原生依赖 |
| blessed 系 | ✅ 原生 scrollable/table 控件 | ❌ 命令式、原版无维护 | ◐ | 纯 JS |

## 决策

选 **Ink**。这是一次**权衡后的明确选择**，而非「安全默认」：

- 取其**生态成熟度与 React DX**（降低长期维护与协作成本）。
- Ink 最致命的场景（渲染上万行）**被本项目的分页架构从源头禁止**——网格任意时刻只渲染一个视口（约 50 行）。
  因此 30 FPS 上限的残余风险仅限「滚动/逐格编辑视口内 ~50 行」的每帧延迟，可用工程手段压住。

## 强制性缓解策略（写进实现规范）

1. **严格分页 + 游标**：DB 侧分页，永不全量进内存（架构第 4、12 节）。
2. **视口虚拟化（windowing）**：用 `useStdoutDimensions()` 算可见行数，只渲染 `visible + buffer`。
3. **`<Static>` 用于滚动历史/日志**：不可变内容走 Ink 的虚拟列表路径，不参与重绘。
4. **激进 memoization**：DataGrid 拆为紧致独立组件，单元格 `memo` 化；状态分片，避免全树重渲染。
5. **输入去抖/节流**：快速滚动时合并帧，规避 30 FPS 上限下的积压。

## 撤退路径（推翻条件）

若 Phase 1/2 实测中，视口内滚动/编辑延迟在上述缓解后仍不可接受：
→ 切换至 **OpenTUI**。代价被架构控制在 `presentation/` 层（domain/application 不变），
　 数据网格组件按新框架重写，其余链路不动。这正是「分层隔离 TUI 框架」的价值兑现。
